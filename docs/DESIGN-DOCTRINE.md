# Дизайн-доктрина MedBook

Референс: главный экран Payme (super-app паттерн). Этот документ — правила, по
которым собирается и полируется **каждый** экран проекта: Mini App (главный
носитель), CRM, кабинет врача. При ревью экрана прогоняйте чек-лист внизу.

## Как называется подход

Это не один приём, а связка именованных паттернов поверх одной философии —
**state-first (anticipatory) design**: интерфейс предугадывает текущий вопрос
пользователя и отвечает на него до первого тапа, вместо того чтобы показывать
карту разделов.

| Паттерн | Что это | Где у Payme |
|---|---|---|
| Glanceable dashboard | Ответ виден за <1 сек без действий | Баланс 36px первой строкой |
| Bento grid | Асимметричная сетка: размер = важность | Круг уведомлений, высокая карта просрочки, мелкие сервисы |
| Semantic color | Цвет кодирует статус, не украшает | Лососёвая карта «платежи просрочены» |
| Thumb-zone ergonomics | Главные действия в зоне большого пальца | Пилюли «Оплатить/Перевести/Scan&GO», нижняя шторка |
| Progressive disclosure | Слои: подсказка → жест → полный экран | Пикающая шторка «свайпните для оплаты» |
| Conversational microcopy | Тексты говорят как человек | «Я потратила на образ невесты…» |
| Object-as-card | Сущность = карточка со статусом внутри | Каждый блок — живой объект, не пункт меню |

## Почему это выигрывает (механика)

1. **Закон Хика** — меньше равнозначных вариантов → быстрее решение. Меню из
   8 одинаковых плиток заставляет думать; одна hero-карта + 2 действия — нет.
2. **Закон Фиттса** — большие цели внизу экрана достигаются быстрее. Верхние
   углы — мёртвая зона на 6.1–6.7".
3. **Преаттентивное восприятие** — цвет и размер мозг считывает до чтения.
   Лососёвая карта «просрочено» работает за 13 мс, бейдж с цифрой — нет.
4. **Zero-tap value** — польза до первого действия формирует привычку
   открывать апп «просто глянуть».
5. **Эмоциональный дизайн** — живой язык + отклик (haptic, motion) = доверие.
   Пользователь прощает ошибки аппу, который «разговаривает».

## Правила

Каждое правило — формулировка + практический тест. Тест провалился → экран
не готов.

**1. Экран отвечает на вопрос, а не показывает структуру.**
Первый экран — текущее состояние пользователя (ближайший визит, очередь,
готовые результаты), не каталог разделов.
*Тест: что пользователь узнал за 0 тапов? Если ничего — экран провален.*

**2. Размер = важность.**
Никаких сеток из одинаковых плиток. Если всё одного размера — решение о
приоритете не принято.
*Тест: закрой текст — по одним габаритам видно, что главное?*

**3. Цвет — это статус.**
Лосось/красный = требует действия, зелёный = готово/успех, акцент = инфо/
выбрано, hint = спокойно. Цвет не используется «для красоты».
*Тест: каждый цветной элемент можно назвать словом-статусом?*

**4. Главное действие — под большим пальцем.**
Primary CTA, таб-бар, шторки — нижняя треть. Верх — только информация.
*Тест: дотянешься до главного действия одной рукой на 6.7"?*

**5. Жест для частого, кнопка для редкого.**
Частое ежедневное действие получает жест (свайп-талон) + текстовую
аффордансу «свайпните вверх…». Жест всегда дублируется тапом.
*Тест: у жеста есть подсказка и кнопочный дубль?*

**6. Тексты говорят, как человек.**
Время суток в приветствии, обращение, разговорные фразы: «перед вами 2
человека — успеете выпить чай», «Запись отменена. Бывает». Без канцелярита
(«операция выполнена успешно» — запрещено). Все строки RU+UZ (Dict parity).
*Тест: фразу можно произнести вслух другу без неловкости?*

**7. Каждое состояние спроектировано.**
Loading = скелетон той же геометрии (без прыжка layout). Empty = тёплый текст
+ действие, не тупик. Error = что случилось + что делать.
*Тест: у экрана отрисованы все 4 состояния (data/loading/empty/error)?*

**8. Движение объясняет причинность.**
Анимация показывает, откуда пришло и куда ушло (шторка — снизу, удаление —
вниз). 200–400 мс, spring-кривые, prefers-reduced-motion уважается. Декор без
смысла (бесконечные блёстки) — нет.
*Тест: можешь сказать, ЧТО объясняет каждая анимация?*

**9. Живое обновляется само.**
Очередь, статусы, чат — SSE/optimistic. Никаких «потяните для обновления»
как единственного пути. Реальное время помечается живой точкой-пульсом.
*Тест: данные изменились на другой поверхности — экран узнал без действий?*

**10. Карточка = объект со статусом.**
Любая сущность (запись, анализ, счёт) рендерится карточкой, внутри которой
виден её статус и следующее действие. Тап = детали, без сюрпризов.
*Тест: статус объекта виден без открытия деталей?*

**11. Не больше трёх главных действий на экран.**
Остальное — за «ещё», в шторке или на внутренних экранах.
*Тест: посчитай CTA — их ≤3?*

**12. Данные внутри навигации.**
Плитка-раздел показывает превью содержимого: «Анализы · последний 10 июня»,
а не голую иконку. Навигация — это уже информация.
*Тест: плитка полезна, даже если по ней не тапнуть?*

**13. Празднуй завершения.**
Успех (запись создана, оплата прошла) получает момент: haptic success +
анимация-отметка. Один раз, без повторов на ремаунте.
*Тест: финал сценария отличается по ощущению от рядового перехода?*

**14. Тёмная тема — отдельный дизайн, не инверсия.**
Токены через light-dark(), тени заменяются ring+deep-shadow, контраст
статусов проверяется в обеих темах. (См. mini-app-tokens.ts.)
*Тест: скриншот в тёмной — все статусы различимы?*

**15. Слои глубины консистентны.**
Атмосфера (аврора) → контент-карты → шторки/оверлеи → тосты. Z-индексы и
блюры не назначаются ситуативно.
*Тест: новый оверлей встал в существующую шкалу слоёв?*

## Чек-лист ревью экрана (быстрый)

- [ ] Ответ за 0 тапов есть (П1)
- [ ] Иерархия размеров отражает важность (П2)
- [ ] Цвета = статусы, читаются в обеих темах (П3, П14)
- [ ] Primary CTA в нижней трети, ≤3 действий (П4, П11)
- [ ] Все 4 состояния отрисованы, скелетон без прыжка (П7)
- [ ] Тексты живые, RU+UZ (П6)
- [ ] Анимации объясняют причинность, reduced-motion ок (П8)
- [ ] Live-данные обновляются сами (П9)

## Применение по поверхностям

- **Mini App** — полный объём доктрины; hero-зона главной = машина состояний
  (очередь → визит сегодня → скоро → результаты → тишина).
- **CRM** (feature-freeze, polish-only) — правила применяются при полировке:
  статусные цвета, состояния экранов, размерная иерархия в дашбордах.
- **Кабинет врача** — то же при возобновлении работ (см. _ROADMAP.md).
