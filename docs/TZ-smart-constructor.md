# ТЗ — Умный конструктор приёма v2 (без AI)

> Составлено: 2026-06-10. Источник: аудит конструктора (reception → VisitNote →
> печать → доставка пациенту) + ревизия schema.prisma. Дополняет
> `docs/TZ-doctor-cabinet.md` (P0–P3 выполнены и задеплоены). AI-слой (Phase 3b
> роадмапа) сознательно **вне scope** — всё ниже детерминировано.

## 0. Контекст и цель

Сегодня конструктор — быстрый блокнот: 5 полей-чипов свободного текста
(`VisitNote.complaints/anamnesis/examination/prescriptions/advice`,
`schema.prisma:1883-1887`), ICD-10 автокомплит, CDS-предупреждения, markdown-
заключение + памятка, печать через Cmd+P, PDF пациенту в Mini App.

При этом в схеме УЖЕ есть незадействованный фундамент:

| Модель | Где | Что умеет | Чего не хватает |
|---|---|---|---|
| `Drug` + `DrugBrand` + `DrugInteraction` | `schema.prisma:887,969,945` | МНН+бренды, формы/дозировки (`forms` Json), `indications` (ICD-префиксы!), `defaultDosing`, противопоказания, побочки, beremennost | конструктор назначений их **не использует** — назначения пишутся строками |
| `ClinicalProtocol` | `schema.prisma:992` | «Применить стандарт» по ICD-префиксу: чипы всех секций + назначения + лабы + markdown | назначения в протоколе — free-text строки; протоколы только глобальные, клиника/врач не могут создавать свои |
| `HandoutTemplate` | `schema.prisma:1041` | библиотека памяток по ICD-префиксам | только RU (`bodyMd` без UZ), вставка ручная через drawer — само ничего не выходит |
| `ClinicCatalogOverlay` | `schema.prisma:1088` | скрыть глобальную запись | `overridesJson` зарезервирован (G8), **правка не реализована** — «менять от руки» нельзя |
| `Prescription` + `MedicationReminderSend` + Mini App дашборд | `schema.prisma:2644,2776`, `/api/miniapp/medications` | курс лекарств у пациента в телефоне, напоминания, snooze — **уже работает** (Phase 16 W3) | из конструктора туда ничего не попадает; `drugName/dosage` — free-text, `caseId` обязателен |
| `PatientAllergy/ChronicCondition/Diagnosis` | `schema.prisma:2682-2741` | структурная история пациента | конструктор их не показывает рядом и не пополняет в один клик |

**Цель:** врач выбирает диагноз — само выходит «что делать / уход / режим /
когда срочно к врачу» и предлагается стандарт лечения; выбирает препарат —
само выходит схема «как принимать» с дозировкой; всё правится руками; все
справочники (препараты, диагнозы, памятки, протоколы) редактируемы и
пополняемы клиникой и врачом; на выходе — документ, за который не стыдно,
и курс лекарств, который сам оказывается в телефоне пациента.

**Продуктовый ориентир («ахуели»):** пациент ещё не дошёл до ресепшена, а в
телефоне уже памятка, расписание таблеток с напоминаниями и дата контрольного
визита; на бумаге — фирменный бланк с сеткой приёма и QR подлинности.

## 1. Scope

**In:** база знаний диагнозов, структурные назначения на каталоге `Drug`,
протоколы v2 (клиничные + личные), редактируемые справочники, печать v2
(бланк/номер/QR/сетка приёма/пакет), мост в Mini App-напоминания, follow-up,
copy-forward повторного визита с динамикой и диффом лечения, body map.

**Out:** любой AI/LLM (генерация, подсказки, диктовка) — Phase 3b; RxNorm и
внешние формуляры; интеграции с аптеками/МоЗ; пересмотр e-рецептов (`EPrescription`
остаётся как есть, печать пакетом его включает).

## 2. Приоритеты

| Фаза | Тема | Почему | Объём |
|---|---|---|---|
| **Ф0** | Гигиена финализации + бланк/номер | пустые визиты = юридическая дыра; быстрые победы | S |
| **Ф1** | База знаний диагнозов (гайды) | главная хотелка: «выбрал болезнь — само вышло» | M |
| **Ф2** | Структурные назначения на `Drug` | фундамент: CDS-надёжность, сетка, напоминания, дифф | L |
| **Ф3** | Протоколы v2 + «сохранить приём как протокол» | one-click оформление визита | M |
| **Ф4** | Редактируемые справочники | «менять и добавлять всё-всё-всё от руки» | M |
| **Ф5** | Печать v2 + QR-верификация | лицо клиники + анти-подделка | M |
| **Ф6** | Петля лечения → телефон пациента | вау-эффект, конкурентов с этим нет | M |
| **Ф7** | Повторный визит: copy-forward + динамика | время врача на хрониках (неврология!) | M |
| **Ф8** | Body map | визуальный вау для невролога, опционально | S–M |

## Ф0 — Гигиена финализации + якоря документа

### Ф0.1 Гейт финализации

**Проблема:** `POST /api/crm/visit-notes/[id]/finalize` не проверяет контент —
можно финализировать пустую заметку без диагноза.

**Решение:** в finalize-роуте: нет `diagnosisCode` → `400 {error:"DIAGNOSIS_REQUIRED"}`
(жёстко). Пустые `complaints`/`advice`/handout — НЕ блокируют, но UI перед
финализацией показывает confirm-диалог со списком пустых секций («Финализировать
без жалоб и памятки?»). Кнопка в `active-patient-card.tsx` дизейблится без диагноза
с тултипом.

**Acceptance:** финализация без диагноза невозможна (API+UI); финализация с
пустыми секциями требует явного подтверждения.

### Ф0.2 Бланк, номер, подпись

**Решение:**
1. `Clinic.letterheadUrl String?` — шапка-изображение для печати (фолбэк —
   текущий текстовый хедер из print-роута).
2. Нумерация: `model DocumentCounter { clinicId, year Int, kind String, value Int,
   @@unique([clinicId, year, kind]) }`; на финализации атомарный инкремент →
   `Document.number` вида `NF-2026-000123` (slug клиники). Печатается в шапке.
3. Блок подписи: «Врач: {ФИО, специализация} ____________» + дата.

**Acceptance:** заключение печатается на бланке с номером и блоком подписи;
номера монотонны в рамках клиника+год.

## Ф1 — База знаний диагнозов («выбрал болезнь — само вышло»)

**Проблема:** при выборе ICD-кода врач получает только код+название. Контент
«что делать / уход / когда срочно» живёт в `HandoutTemplate`, но его нужно
искать руками в drawer'е; он RU-only и не структурирован.

**Решение:**

1. **Schema — новая модель** (структурные блоки, не один blob — ради печати
   по секциям, частичной вставки и RU/UZ паритета):
   ```prisma
   model DiagnosisGuide {
     id          String  @id @default(cuid())
     clinicId    String? // null = глобальный сид; клиника может создать свой
     code        String  // slug для overlay/favorites ("g43-migraine")
     matchPrefix String  // ICD-10 префикс; длиннейший префикс выигрывает (как ClinicalProtocol:988)
     titleRu     String
     titleUz     String?
     whatToDoRu  String? @db.Text   // «Что делать» — план для пациента
     whatToDoUz  String? @db.Text
     careRu      String? @db.Text   // «Уход и режим»
     careUz      String? @db.Text
     lifestyleRu String? @db.Text   // «Образ жизни / диета»
     lifestyleUz String? @db.Text
     redFlagsRu  String? @db.Text   // «Срочно к врачу, если…»
     redFlagsUz  String? @db.Text
     adviceChips String[] @default([]) // готовые чипы в поле advice
     defaultFollowUpDays Int?
     sortOrder   Int     @default(0)
     active      Boolean @default(true)
     createdAt   DateTime @default(now())
     updatedAt   DateTime @updatedAt
     @@unique([clinicId, code])
     @@index([matchPrefix, active])
   }
   ```
   `CatalogEntityType += GUIDE` (для overlay и избранного).
2. **API:** `GET /api/crm/guides?icd=G43.0` — подбор по префиксу (точный код →
   3-значная категория), с учётом overlay клиники; CRUD в Ф4.
3. **UI (reception):** при установке `diagnosisCode` в `structured-fields-panel.tsx`
   под диагнозом появляется карточка «База знаний: {титул}» с секциями-аккордеоном
   и кнопками: «чип → в советы» (adviceChips), «Вставить в памятку» (секция →
   handout-markdown), «Вставить всё». После вставки — обычный текст,
   **правится руками** как сейчас (autosave не трогаем).
4. **Композер v2:** `composePatientHandoutRu()` (`src/lib/catalogs/handout-composer.ts:59-155`)
   → `composePatientHandout(locale)`: жалобы/советы + блоки гайда (что делать →
   уход → red flags) + назначения (Ф2) + дата контроля (Ф6). UZ-вариант
   обязателен (паритет ключей уже enforced vitest'ом).
5. **Сид:** `scripts/seed-knowledge.ts` — топ-50 неврологических диагнозов
   (G43/G40/G45/G47/M54/G62/R51/F41/I63/G35…) RU+UZ. Контент готовится вне
   продукта и правится клиникой в Ф4 (runtime-AI нет).

**Acceptance:** выбор G43.0 мгновенно показывает гайд; «Вставить всё» наполняет
советы и памятку; вставленное редактируется; узбекская печать берёт Uz-блоки.

## Ф2 — Структурные назначения на каталоге Drug (фундамент)

**Проблема:** `VisitNote.prescriptions String[]` (`schema.prisma:1886`) — голые
строки. Следствия: CDS матчит по тексту (опечатка → промолчит), печать не может
построить расписание, в напоминания пациенту ничего не летит, дифф лечения
невозможен. При этом каталог `Drug` с `forms`, `defaultDosing`, `indications`
и брендами (`DrugBrand` — поиск «Конкор» → bisoprolol) уже существует и
заполняется — конструктор его просто не использует.

**Решение:**

1. **Schema:**
   ```prisma
   enum MealRelation { BEFORE_MEAL WITH_MEAL AFTER_MEAL EMPTY_STOMACH NO_MATTER }

   model VisitPrescription {
     id            String    @id @default(cuid())
     clinicId      String
     visitNoteId   String
     visitNote     VisitNote @relation(fields: [visitNoteId], references: [id], onDelete: Cascade)
     drugId        String?   // null = ручной ввод вне каталога (fallback обязателен!)
     drug          Drug?     @relation(fields: [drugId], references: [id])
     displayName   String    // снапшот («Конкор (бисопролол)») — печать стабильна при правках каталога
     form          String?   // из Drug.forms
     strength      String?   // "5 мг"
     dose          String    // "1 таб", "10 мл"
     timesOfDay    String[]  // MORNING|NOON|EVENING|NIGHT
     mealRelation  MealRelation @default(NO_MATTER)
     durationDays  Int?
     instructionRu String?   @db.Text // авто из Drug.defaultDosing/гайда → правится рукой
     instructionUz String?   @db.Text
     remindPatient Boolean   @default(true) // мостить в напоминания (Ф6)
     sortOrder     Int       @default(0)
     createdAt     DateTime  @default(now())
     updatedAt     DateTime  @updatedAt
     @@index([clinicId, visitNoteId])
   }
   ```
   `VisitNote.prescriptions` остаётся read-only legacy: старые заметки рендерят
   чипы, новые пишут строки `VisitPrescription`. Данные не мигрируем.
2. **UI — конструктор схемы** вместо чип-инпута в `structured-fields-panel.tsx`:
   поиск по каталогу (бренд/МНН/ATC, top-12 как ICD), при выборе авто-подстановка:
   форма/дозировка из `Drug.forms`, схема из `Drug.defaultDosing.adult`,
   «как принимать» в `instructionRu/Uz`; сегмент-контролы утро/день/вечер/ночь,
   до/во время/после еды, длительность. Кнопка «Свой препарат» — свободный ввод
   (только `displayName`+`dose`+схема). Всё редактируется после подстановки.
3. **Диагноз-подсказки:** под полем — «Часто назначают при {диагноз}»: выборка
   `Drug.indications` по ICD-префиксу (механизм задуман в комменте `schema.prisma:906-908`,
   Phase G2 — реализуем).
4. **CDS v2:** `useCdsDrugCheck` переводим с текста на пары `drugId` →
   `DrugInteraction` (надёжно) + `PatientAllergy.substance` матч по МНН/бренду;
   freeText-позиции продолжают текстовый матч (как сейчас). Аудит overrides не меняется.
5. **API:** `PATCH /api/crm/visit-notes/[id]` принимает `visitPrescriptions[]`
   (replace-all в транзакции — консистентно с autosave-моделью); print/composer
   читают строки.

**Acceptance:** «Конк» находит Конкор→бисопролол; выбор заполняет схему и
инструкцию; опечатки невозможны для каталожных позиций; CDS ловит пару по id;
свободный ввод остаётся; старые заметки не ломаются.

## Ф3 — Протоколы v2 + «Сохранить приём как протокол»

**Проблема:** `ClinicalProtocol` (`schema.prisma:992`) уже умеет «Применить
стандарт», но: назначения в нём — строки (`prescriptionsTemplate:1016`),
протоколы только глобальные (клиника и врач не могут создать свои).

**Решение:**

1. **Schema:** `ClinicalProtocol += prescriptionItems Json?` — массив
   `{drugId?, displayName, dose, timesOfDay[], mealRelation, durationDays}`;
   `+= clinicId String?` и `+= doctorId String?` (null/null = глобальный,
   clinic-only, личный врача); `+= guideCode String?` (связка с гайдом Ф1);
   `+= followUpDays Int?`. `prescriptionsTemplate` — legacy fallback.
2. **«Применить стандарт» v2:** заполняет чипы секций (как сейчас) + строки
   `VisitPrescription` из `prescriptionItems` + advice + черновик `LabOrder`
   из `recommendedLabs` + followUpDays. Порядок выбора при нескольких
   совпадениях: личный врача → клиничный → глобальный, внутри — длиннейший префикс.
3. **Киллер-фича «Сохранить как протокол»:** кнопка в reception — текущее
   состояние заметки (чипы+назначения+советы+контроль) сохраняется как личный
   протокол врача с именем. Один раз оформил мигрень — дальше каждый такой
   приём = один клик + правки. Аудит создания.
4. **API:** `GET/POST/PATCH /api/crm/protocols` (DOCTOR — личные; ADMIN — клиничные),
   подбор по ICD — расширение существующего endpoint'а применения.

**Acceptance:** применение протокола раскладывает ВЕСЬ визит включая структурные
назначения; врач сохраняет приём как личный протокол и переиспользует; приоритет
личный>клиника>глобальный работает.

## Ф4 — Редактируемые справочники («всё-всё-всё от руки»)

**Проблема:** каталоги глобальные read-only; `ClinicCatalogOverlay.overridesJson`
зарезервирован (`schema.prisma:1097-1099`), но не реализован — клиника не может
ни поправить дозировку, ни добавить свой препарат/гайд/памятку.

**Решение:**

1. **Overlay v2:** реализовать `overridesJson` — JSON-patch поверх глобальной
   записи (своё `defaultDosing`, свой `instructionRu`, переименование, свои
   `adviceChips`). Применяется при чтении каталога (один helper
   `applyClinicOverlay(entity, overlay)` в `src/server/catalog/`).
2. **Локальные записи:** `Drug`, `DiagnosisGuide`, `ClinicalProtocol`,
   `HandoutTemplate` получают `clinicId String?` (у Drug — добавить; у Guide
   есть с Ф1). Поиск = глобальные (минус скрытые overlay) + клиничные.
3. **UI «База знаний»** — новый раздел `/crm/settings/knowledge` (ADMIN):
   табы Препараты / Диагнозы-гайды / Протоколы / Памятки; таблица + редактор
   (формы по блокам, RU/UZ рядом, превью печати). Врач из кабинета: правка
   ЛИЧНЫХ протоколов + предложение правки в клиничную базу (просто deep-link
   в настройки, без модерационного флоу).
4. **HandoutTemplate += bodyMdUz** — закрыть UZ-дыру библиотеки памяток.
5. **Аудит:** каждый write — `audit()` (`KNOWLEDGE_*` actions), как везде.

**Acceptance:** админ меняет дозировку каталожного препарата — у врачей
подставляется правленая; клиника добавляет свой препарат и гайд — ищутся наравне
с глобальными; всё аудируется.

## Ф5 — Печать v2 + QR-верификация

**Проблема:** печать — голый HTML через Cmd+P (`/api/crm/visit-notes/[id]/print`),
без бланка, подписи, расписания приёма и защиты от подделки. Композер RU-only.

**Решение:**

1. **Сетка приёма лекарств** (главный визуальный вау на бумаге): таблица
   препараты × (утро/день/вечер/ночь) с дозой в ячейке, колонка «еда» и
   «длительность» — рендер из `VisitPrescription` (детерминированный компонент
   в `render-handout.ts`, общий для print-роута и PDF-worker'а).
2. **QR-верификация:** `Document += verifyToken String? @unique` (генерится
   worker'ом на финализации); публичная страница `/v/[token]` — «Документ
   подлинный: {клиника}, врач {ФИО}, {дата}, № {number}» + инициалы пациента.
   **Без PII и без контента** (приватность). QR с этой ссылкой — в подвале
   заключения, рецепта, направления. Анти-подделка больничных/направлений.
3. **Двуязычная печать:** `?lang` по умолчанию из `Patient.preferredLang`;
   композер v2 (Ф1) + Uz-блоки; переключатель на печатной странице.
4. **Пакетная печать:** `?type=package` — заключение + памятка с сеткой +
   e-рецепты + направления одним документом с разрывами страниц («распечатать
   всё» одной кнопкой после финализации).
5. Бланк/номер из Ф0 применяются ко всем типам.

**Acceptance:** печать на бланке с номером, сеткой приёма и QR; скан QR
показывает подлинность без PII; UZ-пациент получает узбекскую памятку; одна
кнопка печатает весь пакет.

## Ф6 — Петля лечения → телефон пациента

**Проблема:** Mini App-дашборд лекарств с напоминаниями УЖЕ работает
(`/api/miniapp/medications`, `MedicationReminderSend`, snooze, расчёт следующего
приёма в TZ клиники) — но кормится только ручными `Prescription` из CRM-кейсов.
Из конструктора туда не летит ничего. `Prescription.caseId` обязателен
(`schema.prisma:2648`).

**Решение:**

1. **Schema:** `Prescription.caseId String?` (optional) + `Prescription.visitNoteId String?`
   + `@@unique([visitNoteId, drugKey])`-стратегия идемпотентности (см. п.2).
2. **Мост в worker'е финализации** (`src/server/workers/visit-note-handout.ts`,
   тот же sweep): для каждой `VisitPrescription` с `remindPatient=true` —
   upsert `Prescription { drugName: displayName, dosage: dose+strength,
   schedule: {times: маппинг timesOfDay→часы клиники, days: durationDays},
   status: ACTIVE, remindersEnabled: clinic.medicationRemindersEnabled }`.
   Идемпотентность по `visitNoteId+sortOrder`. Маппинг timesOfDay→времена —
   настройка клиники (дефолт 08:00/13:00/19:00/22:00).
3. **Follow-up:** `VisitNote += followUpDays Int?, followUpNote String?`
   (преднаполняется из гайда/протокола). На финализации: дата печатается в
   заключении; в Mini App карточка прошедшего визита получает кнопку
   «Записаться на контроль ~{дата}» (deep-link в booking); создаётся `Action`
   ресепшену «позвать на контроль» с due-датой.
4. **События:** worker публикует invalidation `['miniapp','medications']`
   (паттерн как `visit-note.finalized` → documents).

**Acceptance:** врач финализирует → у пациента в Mini App курс с напоминаниями
без единого действия ресепшена; контрольный визит — на бумаге, в Mini App и в
задачах ресепшена.

---

## Ф7 — Повторный визит: copy-forward + динамика + дифф лечения

**Проблема:** повторный визит = чистый лист; хроники (вся неврология)
перепечатываются. Динамика состояния нигде не фиксируется.

**Решение:**

1. **Copy-forward:** кнопка «Продолжить от прошлого визита» (виден последний
   FINALIZED `VisitNote` пациента у этого врача): копирует диагноз, чипы
   anamnesis/complaints (с визуальной пометкой «из прошлого визита» до первого
   редактирования), структурные назначения. Не автоматom — врач решает.
2. **Динамика:** `VisitNote += dynamics String?` (`IMPROVED|STABLE|WORSE`) +
   `dynamicsNote String?` — сегмент-контрол в шапке сессии при наличии прошлого
   визита. Печатается строкой «Динамика: улучшение» в заключении.
3. **Дифф лечения** (возможен только благодаря Ф2): детерминированное сравнение
   `VisitPrescription` нового визита с прошлым → блок «Изменения в лечении:
   ↑ доза X 5→10 мг · отменён Y · добавлен Z» — в заключении и памятке. Для
   хроника и его родственников это самая ценная строка документа.
4. **Один клик в историю:** с диагноза — «в хронические» (`PatientChronicCondition`),
   с финализации — upsert `PatientDiagnosis` (ACTIVE), c аллергии в CDS-карточке —
   «записать аллергию» (`PatientAllergy`). Карточка пациента наполняется сама.

**Acceptance:** повторный приём мигрени оформляется правкой копии за <1 мин;
дифф лечения печатается; хроника/диагнозы/аллергии пополняются из приёма в один клик.

## Ф8 — Body map (опциональный вау)

**Решение:** `VisitNote += bodyMap Json?` — точки `{x, y, view: FRONT|BACK,
label?}` на SVG-фигуре человека; компонент в секции осмотра (клик — точка,
клик по точке — подпись/удалить); рендер пинов на печатной форме осмотра.
Неврологический кейс: зоны боли/парестезий/триггерные точки. Без дерматомных
оверлеев в v1.

**Acceptance:** врач ставит точки, они сохраняются, печатаются и видны при
copy-forward следующего визита.

## 6. Последовательность (рекомендуемая)

1. **Ф0** — мелко, сразу закрывает юридическую дыру.
2. **Ф1** — главная хотелка, видимый эффект за одну фазу (не зависит от Ф2).
3. **Ф2** — самый тяжёлый фундамент; всё вау ниже стоит на нём.
4. **Ф3 → Ф5 → Ф6** — one-click визит, затем красивая бумага, затем мост в телефон
   (порядок выбран так, чтобы каждый деплой давал демонстрируемый эффект).
5. **Ф4** — редактируемость (можно тянуть параллельно с Ф5/Ф6 — другие файлы).
6. **Ф7 → Ф8** — повторные визиты и body map.
