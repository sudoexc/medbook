# ТЗ — Two Lanes: записи ⊥ живая очередь

**Статус:** Approved 2026-07-03 (решение владельца) · заменяет serveAt EDF модель
**Скоуп:** развод записей (slot-based) и живой очереди (FIFO) на два полностью независимых потока на всех поверхностях: CRM (ресепшн, appointments, календарь), кабинет врача, mini-app, киоск, ТВ, публичный статус талона.
**Связанные:** `docs/TZ.md` §6, `src/lib/queue-ordering.ts` (старая EDF-модель), commit `0f036cb` (serveAt EDF — отменяется этим ТЗ).

---

## 0. TL;DR — целевая модель

Сегодня `serveAtMs = channel==='WALKIN' ? queuedAt : max(slot, queuedAt)` вмешивает
брони в живую очередь, а `maxBookableSlotsPerDay` режет сетку слотов «в пользу»
очереди. Обе связки удаляются:

1. **Записи (SCHEDULE lane)** — чисто расписание. Слоты = вся сетка графика
   врача (шаг 20 мин) минус time-off минус пересечения бронь-vs-бронь.
   **Лимит `maxBookableSlotsPerDay` не применяется** — записываться можно
   сколько угодно в рамках графика. Walk-in'ы слоты НЕ блокируют.
2. **Живая очередь (LIVE lane)** — только walk-in'ы (`channel === WALKIN`).
   Чистый FIFO: `queuePriority DESC → queuedAt ASC → ticketSeq ASC`.
   **Времени у очереди нет**: слот-время строки никогда не влияет на порядок.
   Бронь на 13:00 никак не мешает walk-in'у, пришедшему в 13:00, и наоборот.
3. **Врач/ресепшн выбирает сам** — две полосы рядом, на каждой строке своя
   кнопка: «Вызвать» (очередь) / «Начать» (запись). Никакого автоматического
   интерливинга, никакого «бронь всплывает в очереди, когда пришло её время».

Пришедшая запись («Пришёл», киоск-чекин) — это **arrived-состояние внутри
SCHEDULE lane** (queueStatus=WAITING + талон как идентификатор), но в порядок
LIVE lane она не встаёт и позицию/ETA очереди не получает.

## 1. Инварианты

- I1. Порядок LIVE lane зависит только от `(queuePriority, queuedAt, ticketSeq)`.
  `Appointment.date/time` walk-in-строки — техническое поле (календарь/день),
  на порядок не влияет.
- I2. Строка `channel !== WALKIN` никогда не входит в список waiting LIVE lane
  ни на одной поверхности (проекция, ресепшн, ТВ, киоск, статус талона).
- I3. `current` (на приёме) — один на врача, из ЛЮБОЙ полосы: врач мог взять
  и walk-in, и запись.
- I4. Слот-генерация не читает `maxBookableSlotsPerDay` и не считает walk-in
  строки блокерами.
- I5. Тикеты — идентификаторы, не позиции: счётчик `queueOrder/ticketSeq`
  общий на день (как сейчас), печать/оверлей работают для обеих полос.
- I6. Переходы статусов не меняются (`appointment-transitions.ts`), «Начать»
  по-прежнему требует только отсутствия другого IN_PROGRESS.

## 2. Изменения по слоям

### 2.1 Ядро
- `src/lib/queue-ordering.ts` — `serveAtMs` умирает; `compareQueue` = priority
  DESC → `queuedMs` (queuedAt ?? date fallback) ASC → ticketSeq ASC.
  Добавляется `isLiveLane(a) = a.channel === "WALKIN"` + `splitLanes(rows)`.
- `queue-projection.ts` — waiting = только LIVE lane; current = любой
  IN_PROGRESS. ETA = perVisitMin × позиция в LIVE lane.
- Reorder (`/api/crm/appointments/reorder` + клиентское зеркало) — драгается
  только LIVE lane; slot-floor логика (`max(slot,…)`, `floored[]`) удаляется,
  ответ всегда `exact:true` (форма ответа сохраняется для совместимости).

### 2.2 Чек-ин
- Ресепшн «Пришёл» / киоск-чекин: как сейчас (WAITING + талон + queuedAt),
  НО queuedAt броней больше нигде не участвует в порядке очереди — это только
  метка «ждёт с …». Комменты «serveAt EDF anchor» переписываются.
- `/api/queue/status/[id]` (талон/mini-app): walk-in → позиция+ETA по LIVE
  lane; бронь → `position:null` + слот-время (UI показывает «Ваша запись в
  13:00», не позицию).
- `/ticket/[id]` печать: walk-in → «перед вами N» по LIVE lane; бронь →
  слот-время.

### 2.3 Слоты
- `findAvailableSlots`: cap-логика удаляется целиком; overlap-запрос получает
  `channel != WALKIN` (симметрия с `detectConflicts`/DB EXCLUDE).
- Кабинет врача, настройки: поле `maxBookableSlotsPerDay` из UI удаляется
  (колонка в схеме остаётся, не читается — задокументировано здесь).

### 2.4 UI — две полосы
- **Ресепшн `doctor-queue-panel`**: draggable-середина = только LIVE lane;
  записи — отдельная секция «Записи» (сорт по слоту, Пришёл/Начать, без drag).
- **`doctor-queue-card`**: «Вызвать следующего» → head LIVE lane; записи
  отдельным мини-списком со своим «Начать».
- **`queue-column`**: «Живая очередь» — первичная секция, «Записанные» —
  вторичная; раздельные счётчики.
- **Кабинет врача my-day**: новая карточка «Живая очередь» (live-lane список,
  «Вызвать» на строке) рядом с расписанием (Записи, «Начать» на строке);
  current-card перестаёт автоматически объявлять ближайшую бронь «текущим»
  без выбора врача — показывает её как «Следующая запись» с кнопкой.
- **`/doctor/reception` queue-card**: два раздела вместо смешанного списка.
- **ТВ `/tv/d/[token]`**: уже двухполосный; слева проекция (станет чисто LIVE
  автоматически), справа статус WAITING переименовывается «В очереди» →
  «Пришёл».
- Киоск/публичный борд: без изменений UI (проекция сама станет live-only).

### 2.5 Сиды/тесты
- `_live-queue-seed`: строит обе полосы как сейчас (walk-in'ы → очередь,
  PHONE/TELEGRAM → записи, пришедшие — arrived) — обновить ожидания.
- `queue-ordering.test.ts` → FIFO-спека + lane-split; `reorder-queue-signal`
  → exact-always для walk-in'ов, брони в reorder не участвуют.

## 3. Не делаем
- Не меняем схему БД (кроме — ничего; колонки остаются).
- Не трогаем колл-центр (там очередь звонков), календарь (только брони).
- Не вводим отдельную таблицу очереди — lane = f(channel), этого достаточно.
