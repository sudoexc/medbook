## 🔴 Critical — shipping fake/wrong data

### ✅ C1. Hardcoded fake percentages in tiles — FIXED
- `calendar-tiles.tsx:104` — `yesterdayDelta=8` removed; tile no longer shows fake "+N vs yesterday".
- `patients-tiles.tsx:99` — `noShowPct=5.1` removed; whole risk tile dropped (value AND delta were fake).
- `patients-tiles.tsx:127` — `pct:23` removed; new-week tile keeps real count, no fake delta.
- `patients-tiles.tsx:172` — `pct:12` removed; avg-check tile keeps real money, no fake delta.
- Grid adjusted to `xl:grid-cols-5` (6 → 5 tiles).

### ✅ C2. Doctor cabinets — FALSE POSITIVE
- Re-grounded against current code: doctor API uses `prisma.doctor.findMany({ include: { cabinet: true } })` (route.ts:42) and the detail handler returns the full `Cabinet` record via the same include. There's no `idx + 1` anywhere in `src/app/api/crm/doctors/**` or the seed scripts. Schema has `Doctor.cabinetId String @unique` with strict POST/PATCH validation (`cabinet_required` / `cabinet_taken` / `cabinet_not_found` errors). The original audit claim was wrong.

### ✅ C3. KPI mixes today + 30d windows — FIXED
- The notifications KPI strip used to read `today.sent`, `last30d.delivered`, `today.queued`, `today.failed` — a "delivered" tile silently sliding 30 days of data behind a "today" headline.
- `/api/crm/notifications/stats` now exposes `today.delivered` (counts rows where `deliveredAt >= startOfToday` with status DELIVERED/READ).
- `notifications-kpi-strip.tsx` reads all four numbers from `today.*`; `queued` keeps its realtime semantics (rows in QUEUED state right now) since that's the only window that makes sense for it.
- Hint strings updated in `ru.json` + `uz.json`: delivered=`Сегодня`/`Bugun`, queued=`Сейчас`/`Hozir`.
- The 30d block in `notifications-details-rail.tsx` is unchanged — it's an explicitly-labeled context panel, not part of the headline KPI strip.

### ✅ C4. Clinic settings init-once + currency=UZS — ALREADY DONE / VERIFIED
- `clinic-settings-client.tsx:71-75` has `&& !form` guard — init runs once, refetch doesn't clobber edits.
- Line 157 forces `payload.currency = "UZS"`, line 158 nulls `secondaryCurrency`.
- Line 294 currency input is `readOnly value="UZS"`.
- **Bonus fix**: dropped UTC + Europe/Moscow from the TZ dropdown — clinic is Tashkent-only.

### ✅ C5. Schedule editor data loss — FIXED
Two attack vectors:
1. **Background refetch mid-edit** — `useEffect([doctor.schedules])` blindly reset `slots`. Fix: added `dirtyRef` in `schedule-editor.tsx`; effect skips resync when state is dirty.
2. **Tab switch** — Radix `TabsContent` unmounts inactive by default, wiping local state. Fix: added `forceMount` + `data-[state=inactive]:hidden` to `schedule` AND `services` TabsContent in `doctor-profile-client.tsx` (services editor has the same problem).

### ✅ C6. Billing tiins vs UZS — VERIFIED CORRECT (not a bug)
- `formatTiinsAsUzs(amountTiins)` divides by 100 (tiins → UZS). Called on `BillingPageInvoice.amountTiins` — name + type match.
- `formatDecimalAsUzs(price)` truncates a Decimal(12,2). Called on `subscription.priceMonth` / `plan.priceMonth` — schema confirms `priceMonth Decimal @db.Decimal(12, 2)`.
- Field-naming convention separates them cleanly. Audit was incorrect.

### C7. Other reported critical items (pending verification)

Shipping an editor requires (a) a `RolePermission` Prisma model with per-clinic overrides, (b) seeding it from the static matrix on migration, (c) refactoring ~100 API handlers to consult the DB instead of a literal role list, (d) scope-aware checks (own/all/today), and (e) a bootstrap/lockout guard so a bad save can't kick the last admin out. Doing this safely is a multi-week RBAC project, not an audit-window fix — the existing `readOnlyForAdmin` badge on the page already discloses the constraint. ## 🟡 Major — broken UX

- ✅ Appointments table inert buttons — re-grounded. More menu is actually wired (`RowMenu` calls `onOpen` / `onOpenPatient` / `onCall`); Save lives in `appointment-drawer.tsx` and is wired. The only genuinely inert row action was the Telegram quick-button (`appointments-table.tsx:439-448`) — it only `stopPropagation`-ed. Fixed by routing it to `/${locale}/crm/telegram?patientId=<id>`. To make that URL param actually filter the inbox, added `patientId` end-to-end: server schema (`QueryConversationSchema`), `/api/crm/conversations` `where` clause, and `useConversationsFilters` (URL ⇄ filters ⇄ query string). The patient-quick-actions Telegram button was already navigating to the same URL but the param was a no-op — it works now.
- Right-rail action buttons — re-grounded as false positive. `appointments-right-rail.tsx` "Send reminders" / "Confirm" / "Broadcast" / "Book slot" / overflow-menu items all have real `onClick`/`href` handlers. The reception bottom-row "Применить" redistribute card navigates to the calendar (limited but not inert). No action required.
- ✅ "Merge duplicates" removed from patient-hero.tsx + patient-quick-actions.tsx — the "...Soon" toast was a lie. i18n keys dropped (ru + uz). Real merge needs duplicate detection + audit trail work; restore the menu item with a real handler when that lands.
- ✅ Patient-page SMS quick-send — re-grounded with a fix. The dialog DOES pass `patientId` + `phone` (verified in `patient-card-client.tsx:267-272` + `sms-dialog.tsx:46`). The architectural gap the audit was pointing at: `/api/crm/communications/sms` only wrote a `Communication` row, so quick-sends from the patient page were invisible to the `/crm/sms` inbox (which reads from `Conversation`/`Message`). Fixed by upserting a `Conversation` keyed by `sms:<phoneE164>` (same key as the inbound webhook) and creating a `Message` inside a `$transaction`, then publishing `tg.message.new` so the inbox SSE refreshes. The `Communication` row stays as the per-send audit trail; the new conversation+message is the inbox-visible mirror. Response now includes `conversationId` + `messageId` so callers can deep-link to the thread.
- Locale bleed:
  - ✅ action-center `shortTitleForType` → `shortLabelForRow` reads from `actionCenter.dashboard.tasksQueue.shortLabel.<TYPE>` (RU + UZ keys added)
  - ✅ `.toLocaleString('ru-RU')` calls in CRM (`cases-section`, `doctor-reviews`, `billing-client`, `dsar-review-client`) now route through new `intlLocale(locale)` helper in `lib/format.ts`
  - ✅ English "Loading…" in calendar replaced with `t("calendar.loading")` via a tiny `CalendarLoadingFallback` client component (the `dynamic()` `loading` callback can't `useTranslations` at module scope)
  - ✅ RU-hardcoded toasts in CRM (18 spots / 7 files) — fixed. All `toast.error("…")` / `toast.success("…")` with cyrillic literals now read from `crmToasts.{patient,appointment,case,doctor}.*` (added to ru + uz). The change-password page (`crm/me/change-password`) was 100% RU-bleed (headings, labels, mismatch error, submit button) — fully i18nized under new `meChangePassword` namespace.
  - ✅ missing `/locale` prefix — fixed across 5 surfaces. Three reception widgets (`urgent-alerts-widget.tsx`, `tg-preview-widget.tsx`, `calls-widget.tsx`) use `next/link` so they need `useLocale()` + manual `${locale}` prefix on every `href`. The CRM `error.tsx` boundary + action-center calendar-card link got the same treatment. The settings layout (server component) was switched to the locale-aware `redirect` from `@/i18n/navigation` (`redirect({ href, locale })`). Caveat noted: the codebase has TWO different Link primitives — `next/link` (no auto-prefix) and `@/i18n/navigation` Link (auto-prefixes via `as-needed`). The latter must NEVER be hand-prefixed or it double-prefixes (`/uz/uz/...`); a prior fix to `telegram-page-client.tsx` did exactly that and has been reverted.
- ✅ Destructive actions confirm — services delete (`services-settings-client.tsx`) now wrapped in AlertDialog with the service name in the description body; mutation only fires from `AlertDialogAction.onClick` after explicit confirm. user deactivate transition in `EditUserDialog` already intercepted; DSAR approve ✅

- ✅ TZ dropdown — UTC/Moscow removed, kept `Asia/Tashkent` + `Asia/Samarkand`
- ✅ Integrations: raw JSON textarea for credentials replaced with typed per-provider fields in `integrations-client.tsx` (`PROVIDER_FIELDS` + `VARIANTS` schemas drive `ProviderEditDialog`). Variant dropdown (eskiz/payme/click/uzum/sip/etc) maps to `ProviderConnection.label`; typed inputs build the JSON config. Unknown keys preserved on round-trip so non-schemafied fields aren't dropped. i18n keys added under `settings.integrations.cfg.*` + `variants.*` in ru + uz.
- ✅ Patient cuid manual input — _false positive after re-grounding._ `new-patient-dialog.tsx` collects name/phone/email/dob/gender/source/tags only — no id field. Server auto-allocates per-clinic `patientNumber` via `allocatePatientNumber()` (transactional) and Prisma generates the `id` cuid. No manual cuid surface exists.
