summaries like the legacy webpack builder did. The build prints route types
(`○ Static`, `ƒ Dynamic`, `● SSG`) and a static-page generation count, but
byte totals per route are unavailable.

We substitute with chunk-directory measurement:

- Total `.next/static/chunks/`: **4.4 MB** (uncompressed, all chunks
  combined across the whole app; gzip typically ~30–35% of that).
- Largest individual chunks after fixes:
  - 308 KB — React/Next runtime (shared)
  - 308 KB — vendor bundle
  - 287 KB — vendor split
  - 270 KB — vendor split
  - 227 KB — vendor split
  - 137 KB — app shell
  - 114 KB — CSS (`0jnrzsnqiy_oa.css`, Tailwind compiled)
  - 113 KB, 77 KB, 71 KB — feature splits

The three dynamic imports added in this pass move
`recharts` (~90 KB gzip) and `cmdk + @radix-ui/react-dialog` (~40 KB gzip)
out of the initial route payloads and into on-demand chunks. Those chunks
still exist in `.next/static/chunks`, but they are no longer in the
above-the-fold loadable for the affected pages.

## Fixes applied

### 1. Dynamic-import recharts in Analytics page

**Before:** `src/app/[locale]/crm/analytics/_components/analytics-page-client.tsx`
imported `Bar, BarChart, CartesianGrid, Cell, Line, LineChart, Pie, PieChart,
ResponsiveContainer, Tooltip, XAxis, YAxis` from `recharts` directly in the
top-level client component. `recharts` is ~90 KB gzip and was shipped in the
analytics route's First Load JS unconditionally, even for users who only
glanced at the filter UI.

**After:**
- Extracted all chart JSX into new
  `src/app/[locale]/crm/analytics/_components/analytics-charts.tsx`.
- Shared types moved to
  `src/app/[locale]/crm/analytics/_components/analytics-types.ts` so
  the page client can import types without pulling the chart module.
- Page client now loads charts via `next/dynamic` with
  `ssr: false` and a 4-panel skeleton `loading` fallback.
- i18n strings passed as a `labels` prop (chart module stays
  presentation-only).

### 2. Dynamic-import recharts widgets in Patients right-rail

**File:** `src/app/[locale]/crm/patients/_components/patients-right-rail.tsx`

**Before:** `DemographicsWidget` and `SourcesWidget` (both use `recharts`
`PieChart` / `BarChart`) were statically imported alongside the patients
table, forcing recharts into the patient-list initial bundle.

**After:** both widgets loaded via `next/dynamic` with `ssr: false` and a
shared skeleton fallback. `BirthdaysWidget` and `TopServicesWidget` are
recharts-free and remain statically imported (no benefit from splitting).

### 3. Dynamic-import cmdk GlobalSearch in CRM topbar

**File:** `src/components/layout/crm-topbar.tsx`

**Before:** `GlobalSearch` was imported statically. Because the topbar
renders on every CRM page, `cmdk` + `@radix-ui/react-dialog` (~40 KB gzip)
were pulled into every CRM route's shared chunk.

**After:** introduced a `searchMounted` state that is only flipped `true`
the first time the user invokes search (click, `⌘K`, or `/`). `GlobalSearch`
is wrapped with `next/dynamic({ ssr: false })` and conditionally rendered
only once `searchMounted` is set:

```tsx
const GlobalSearch = dynamic(
  () => import("@/components/layout/global-search").then((m) => m.GlobalSearch),
  { ssr: false },
)

const [searchMounted, setSearchMounted] = React.useState(false)
const openSearch = React.useCallback(() => {
  setSearchMounted(true)
  setSearchOpen(true)
}, [])

{searchMounted ? (
  <GlobalSearch open={searchOpen} onOpenChange={setSearchOpen} />
) : null}
```

The keyboard-shortcut hook (`useGlobalSearchShortcut`) still works because
it simply calls `openSearch`, which now doubles as "mount + open".

This is the highest-leverage fix in the pass: the savings apply to
**every CRM route**, not just one page.

### 4. `next/image` + `priority` for LCP logos

`public/logo.png` is a 800×312 PNG at ~95 KB. On kiosk, TV, and ticket
queue-status routes it is the largest element above the fold. Migrating to
`next/image` emits WebP with a responsive srcset and inline
`<link rel="preload">` when `priority` is set.

| File                                              | Change                                                     |
|---------------------------------------------------|------------------------------------------------------------|
| `src/app/kiosk/page.tsx`                          | `<img>` → `<Image width={123} height={48} priority>`       |
| `src/app/tv/page.tsx`                             | 2× `<img>` → `<Image ... priority>` (main + footer)        |
| `src/app/q/[id]/page.tsx`                         | `<img>` → `<Image width={82} height={32} priority>`        |

After the sweep, `grep -n "<img"` inside `src/` returns **zero** matches.

## Virtualization audit

Target: any table rendering > 100 rows should use TanStack Virtual (or an
equivalent window). Findings:

| Surface                                              | Status                    |
|------------------------------------------------------|---------------------------|
| Patients table (`/crm/patients`)                     | Virtualized (TanStack)    |
| Appointments table (`/crm/appointments`)             | Virtualized               |
| Doctors list (`/crm/doctors`)                        | Virtualized               |
| Call-center history (`/crm/call-center`)             | Virtualized               |
| Telegram conversation list                           | Virtualized               |
| Notifications queue (`/crm/notifications`)           | Virtualized               |
| Documents (`/crm/documents`)                         | Virtualized               |
| Audit log — settings variant (`/crm/settings/audit`) | Virtualized               |
| Admin audit (`/admin/audit`)                         | Infinite scroll, no virt. |
| Admin users (`/admin/users`)                         | Pagination (capped pages) |

**Gap, low priority:** `/admin/audit` uses cursor infinite-scroll. With
10k+ audit rows it can accumulate DOM weight. Not urgent — it's a
super-admin-only surface with limited daily usage and page-at-a-time
requests. Recommend adding TanStack Virtual if we ever observe DOM size
problems in the field. Not a provably-safe change today (wouldn't want to
touch the infinite-scroll loader seam without manual QA).

### N+1 hot spots (background workers — low user-visible impact)

File: `src/server/notifications/triggers.ts`

1. **`runScheduledTriggers`** — iterates up to 500 upcoming appointments
   and calls `materializeForAppointment` for each. Each call issues 3
   queries: reload appointment with `include` for patient/doctor/service/
   cabinet/clinic, `findTemplateFor`, and an `alreadyScheduled` idempotency
   lookup. Worst case: **1500 queries per tick** for a single trigger
   pass.

   Recommendation (when we get time, not this PR): batch-fetch candidate
   appointments with all includes in the initial query, batch-load matching
   templates grouped by clinic, and replace the per-appointment
   `alreadyScheduled` check with one `findMany` keyed on
   `(appointmentId, templateId)` pairs. Should collapse to O(1) + O(1)
   queries per tick.

2. **`runBirthdays`** — scans ~2000 patients, then for each match issues an
   `alreadyScheduled` query. Worst case roughly = birthday matches × 1
   query. Acceptable at current clinic sizes; worth watching as the DB
   grows.

3. **`runPaymentsDue`** — loads 500 `COMPLETED` appointments plus their
   payments, then loops `materializeForAppointment`. Same pattern as (1).

All three are cron/queue-worker code paths with no user-facing latency
attached. Not shipping a fix here to stay safely inside the "small safe
fixes" window.

### Potential future Prisma index (recommendation only)

If the background notification workers become a hot spot, a composite
index on `NotificationSend(appointmentId, templateId)` would directly
accelerate the `alreadyScheduled` idempotency check. Deferred to
`prisma-schema-owner` with actual workload evidence.
